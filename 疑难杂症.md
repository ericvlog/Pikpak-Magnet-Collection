# Vercel 代理复盘：我犯了什么错

## 背景

用户用 Vercel 代理调 PikPak API，登录和刷新 token 正常，但选文件夹和离线（`api-drive.mypikpak.com`）返回 401。

## 我的错误诊断

1. 看到 JWT 里 `iss` 是 `https://pikpak-magnet-collection.vercel.app`，我推断是 PikPak 认证服务器根据 Vercel 的出站 IP 动态设置 issuer，资源服务器校验 issuer 白名单失败 → **这是网络层问题，代理代码改不了**。
2. 我说「大概率无解」，让用户放弃 Vercel 走 Cloudflare Tunnel。

## 实际原因

Vercel 会在请求头里注入大量标头（`x-vercel-*`, `x-forwarded-*` 等）。代理代码只过滤了部分已知的 `x-vercel-` 头，**漏掉了如 `x-forwarded-host`、`trace-id`、`via`、`x-request-id` 等**。PikPak 的某个上游网关读到了这些头中包含的 Vercel 域名，把它当成了 issuer 写入 JWT。

## 正确的修复

把 **所有** 可能泄漏 Vercel 域名的头全部过滤掉：

- 所有 `x-vercel-*` 前缀的头
- 所有 `x-forwarded-*` 前缀的头
- 所有 `x-real-*` 前缀的头
- `trace-id`、`via`、`x-request-id` 等杂项

## 教训

1. **不要过早下「无解」的结论**。用户坚持要试，结果一次就修好了。
2. **代理类问题优先怀疑头泄露**，而不是 IP 侦测。Vercel 作为反向代理，请求头里充满平台信息，漏过滤一个就能让上游服务做出错误判断。
3. **诊断要彻底**：看到 issuer 不对 → 应该先想「PikPak 从哪里拿到这个域名」→ 检查所有转发的请求头 → 找出泄露源。而不是跳过排查直接归因为「IP 层面问题」。

# whatslink.info 图片下载 500 — 过犹不及

## 背景

扩展的 `downloadImage` handler 负责从 whatslink.info 下载图片存为本地 base64。用户反馈 saveImageToLocal 失败，控制台报 500。

## 错误做法

为了让请求「更像浏览器」，我加了大量自定义头：

```
'Origin': urlObj.origin,              // https://whatslink.info
'Sec-Fetch-Site': 'same-origin',      // 撒谎
'Sec-Fetch-Mode': 'cors',
'Sec-Fetch-Dest': 'image',
'Referer': pageUrl || origin + '/',   // 可能传入了 github.io 的 Referer
```

还用 `chrome.cookies.getAll` 手动获取 Cookie 并塞进 `Cookie` 头，同时 `fetch` 还设了 `credentials: 'include'`（等于重复发 Cookie）。

## 实际原因

whatslink.info 的 CDN（Cloudflare）检测到请求头自相矛盾：

1. **Origin 骗人**：`Origin: https://whatslink.info` 但实际请求来自 `chrome-extension://`，Cloudflare 发现 IP/ASN 与 Origin 不匹配，返回 500
2. **Cookie 冲突**：手动 `Cookie` 头 + `credentials: 'include'` 导致重复或过期 Cookie，服务器拒绝
3. **Referer 错误**：Referer 被设成 `https://ericvlog.github.io/...`（页面 URL），whatslink.info 不认识这个域名，可能触发了防盗链
4. **Sec-Fetch 撒谎**：`Sec-Fetch-Site: same-origin` 但实际是 cross-origin，Cloudflare 的安全检查会标记为可疑

## 正确的做法

去掉所有「帮忙」的头，只留下最基本的：

```
User-Agent
Accept
Accept-Language
Referer (固定为 https://whatslink.info/)
Cache-Control / Pragma
```

不要手动 Cookie，不要 Origin，不要 Sec-Fetch-*。扩展后台的 fetch 默认行为（无 CORS 限制、自动处理 Cookie）已经够用，加得越多越容易触发服务器/CDN 的防御机制。

## 教训

**过犹不及**。以为加了更多浏览器头就能骗过服务器，结果每个头都是一个可能的怀疑点。对于图片下载这种简单需求，请求头越精简越好。修复后直接返回 200，图片正常下载。

# whatslink.info 间歇性 520/522 — 源站不稳定

## 症状

- 预览 `https://whatslink.info/api/v1/link?url=...`：第一次可能成功，第二次就 520/522
- 图片 `https://whatslink.info/image/...`：有时 Cloudflare DNS 解析失败，有时无限转圈
- `curl` 从本机测试返回 200，说明**不是全站挂**

## 原因

whatslink.info 的源站服务器不稳定，Cloudflare 到源站经常超时：

- **520**：源站返回了 Cloudflare 看不懂的响应（或空响应）
- **522**：Cloudflare 连不上源站（连接超时）

## 处理方式

代码里加了两样：

1. **重试**：预览遇到 520/522 等非 429 错误，等 1.5s 重试一次
2. **图片下载**：扩展 `downloadImage` 失败 → 页面 fallback fetch → 再失败就保留远程 URL，不阻塞用户

这两项只是让体验不那么差，根因在 whatslink.info 那边，等他们自己修。不从代码层面做更多「优化」了——不是我们的问题。
